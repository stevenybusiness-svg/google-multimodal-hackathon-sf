# Phase 1.5: Live Architecture Visualization - Research

**Researched:** 2026-03-22
**Domain:** Real-time animated node graph visualization (vanilla JS, Canvas/SVG, WebSocket)
**Confidence:** HIGH

## Summary

This phase adds a split-screen live pipeline visualization to the existing Meeting Agent UI. The left side shows an animated node graph of the multimodal pipeline (Mic -> STT -> Gemini -> Dispatcher -> Slack/Calendar/TaskLog, plus Camera -> Vision -> Gemini), with particles flowing along edges and nodes pulsing on activity. The right side shows the existing action output cards. The backend emits pipeline events via a new WebSocket message type on the existing `/ws/audio` connection.

The recommended approach is a **Canvas-based renderer** for the entire pipeline graph (nodes, edges, particles, glow effects). Canvas is the right choice here because: (1) there are only ~10 nodes so performance is trivially adequate, (2) particle animations along edges require per-frame rendering which Canvas excels at, (3) glow/blur effects via `ctx.shadowBlur` are native to Canvas, and (4) no DOM diffing overhead. The project already uses Canvas for the vision overlay, so the pattern is established.

**Primary recommendation:** Single Canvas element for the pipeline graph, driven by `requestAnimationFrame`. Backend emits pipeline events as new message types on the existing `/ws/audio` WebSocket (no new endpoint). Nodes defined as a static layout array with positions, edges as quadratic Bezier curves, particles as small circles moving along those curves. CSS handles the split-screen layout.

**When to use:** When the graph topology is fixed (our pipeline never changes shape).

### Pattern 2: Particle Animation Along Quadratic Bezier Curves
**What:** When data flows through an edge, spawn particles that travel from source node to target node along a curved path. Each particle has a `t` parameter (0 to 1) that advances each frame.
**When to use:** To visualize data flow along edges.
**Example:**
```javascript
// Quadratic Bezier point at parameter t
function bezierPoint(x0, y0, cx, cy, x1, y1, t) {
  const u = 1 - t;
  return {
    x: u * u * x0 + 2 * u * t * cx + t * t * x1,
    y: u * u * y0 + 2 * u * t * cy + t * t * y1,
  };
}

// Particle system
const particles = [];
function spawnParticle(edge, color) {
  particles.push({ edge, t: 0, color, speed: 0.008 + Math.random() * 0.004 });
}

// In animation loop:
for (let i = particles.length - 1; i >= 0; i--) {
  const p = particles[i];
  p.t += p.speed;
  if (p.t >= 1) { particles.splice(i, 1); continue; }
  const pos = bezierPoint(/* source, control, target coords */, p.t);
  ctx.beginPath();
  ctx.arc(pos.x, pos.y, 3, 0, Math.PI * 2);
  ctx.fillStyle = p.color;
  ctx.shadowColor = p.color;
  ctx.shadowBlur = 8;
  ctx.fill();
}
```

### Pattern 3: Node Glow/Pulse via Canvas shadowBlur
**What:** Active nodes get a pulsing glow effect by modulating `shadowBlur` with a sine wave tied to time.
**Example:**
```javascript
function drawNode(ctx, node, time) {
  const isActive = node.activeUntil > time;
  const baseRadius = 24 * (node.size || 1);

  // Glow effect
  if (isActive) {
    const pulse = 0.6 + 0.4 * Math.sin(time * 4); // 4 Hz pulse
    ctx.shadowColor = node.color;
    ctx.shadowBlur = 12 + 8 * pulse;
  } else {
    ctx.shadowColor = 'transparent';
    ctx.shadowBlur = 0;
  }

  // Node circle
  ctx.beginPath();
  ctx.arc(node.x, node.y, baseRadius, 0, Math.PI * 2);
  ctx.fillStyle = isActive ? node.color + '40' : '#161b22';
  ctx.strokeStyle = isActive ? node.color : '#30363d';
  ctx.lineWidth = isActive ? 2.5 : 1.5;
  ctx.fill();
  ctx.stroke();

  // Label
  ctx.shadowBlur = 0;
  ctx.fillStyle = '#c9d1d9';
  ctx.font = 'bold 10px Inter, sans-serif';
  ctx.textAlign = 'center';
  ctx.fillText(node.label, node.x, node.y + baseRadius + 14);
}
```

### Anti-Patterns to Avoid
- **DOM-based particles:** Do NOT create DOM elements for particles -- Canvas is orders of magnitude faster for many small moving elements
- **Recalculating layout each frame:** Node positions are static. Never recompute layout in the animation loop.
- **Heavy shadowBlur on every element:** Canvas shadowBlur is expensive. Use it only on active nodes (typically 1-3 at a time), not all 10+.
- **Blocking the main thread:** Keep animation loop under 2ms per frame. No heavy computation in requestAnimationFrame.
- **Using innerHTML for pipeline UI:** The project explicitly avoids innerHTML (XSS). Use Canvas drawing or textContent only.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Bezier curve math | Custom curve interpolation | Standard quadratic Bezier formula (3 lines) | The math is well-defined; use the formula directly |
| Node layout algorithm | Force-directed or auto-layout | Hard-coded positions | Only ~10 nodes; fixed pipeline topology; manual layout looks better for demos |
| Icon rendering in Canvas | Custom icon drawing | Unicode text or emoji on Canvas, or drawImage with preloaded icon sprites | Canvas text rendering handles icons well enough for a hackathon |
| Split-screen layout | Custom JS layout manager | CSS Flexbox | Already used throughout the project |

**Key insight:** This is a hackathon demo visualization, not a general-purpose graph library. Hard-code everything that can be hard-coded. The pipeline topology is fixed -- there is zero need for dynamic layout algorithms.

## Common Pitfalls

### Pitfall 1: Canvas Blurriness on Retina/HiDPI Displays
**What goes wrong:** Canvas renders at 1x pixel density on retina screens, looking blurry.
**Why it happens:** Canvas `width`/`height` attributes set logical pixels, not device pixels.
**How to avoid:**
```javascript
const dpr = window.devicePixelRatio || 1;
canvas.width = displayWidth * dpr;
canvas.height = displayHeight * dpr;
canvas.style.width = displayWidth + 'px';
canvas.style.height = displayHeight + 'px';
ctx.scale(dpr, dpr);
```
**Warning signs:** Text and circles look fuzzy on MacBook screens.

### Pitfall 2: Canvas Font Loading Race
**What goes wrong:** `ctx.fillText` uses a fallback font because Inter hasn't loaded yet.
**Why it happens:** Canvas doesn't wait for web fonts to load.
**How to avoid:** Use `document.fonts.ready.then(() => startAnimation())` or accept system font fallback (for a hackathon, this is fine).
**Warning signs:** Text renders in serif/default font momentarily.

### Pitfall 3: Animation Loop Accumulating When Tab is Backgrounded
**What goes wrong:** `requestAnimationFrame` pauses when tab is hidden, but timers keep running. On resume, a burst of events fires at once.
**Why it happens:** `setTimeout`/`setInterval` for particle spawning doesn't pause with rAF.
**How to avoid:** Use delta time (`performance.now()`) in the animation loop; clamp deltaTime to max 100ms to prevent jumps.
**Warning signs:** Particles teleport or bunch up after switching tabs.

### Pitfall 4: WebSocket Message Ordering with Pipeline Events
**What goes wrong:** Pipeline events arrive interleaved with existing transcript/action messages, causing confusing animation timing.
**Why it happens:** Backend emits pipeline events alongside existing WS messages.
**How to avoid:** Pipeline events are additive -- they trigger animations independently of existing message handling. The existing handleWsMessage switch already dispatches by type, so adding "pipeline" is safe.
**Warning signs:** None if implemented correctly; the existing message dispatch pattern handles this naturally.

### Pitfall 5: shadowBlur Performance on Many Elements
**What goes wrong:** Frame rate drops below 30fps when multiple nodes have glow effects.
**Why it happens:** Canvas shadowBlur is computationally expensive, especially with large blur radii.
**How to avoid:** Limit active glow to at most 3-4 nodes at once (which naturally happens since pipeline stages activate sequentially). Use modest shadowBlur values (8-16, not 30+). Reset shadowBlur to 0 immediately after drawing glowing elements.
**Warning signs:** Janky animation, especially on lower-end hardware.

### Pitfall 6: Split-Screen Layout Breaking Existing UI
**What goes wrong:** Adding the pipeline view disrupts the existing meeting screen layout.
**Why it happens:** The current meeting screen has transcript (left) + actions (right) layout that needs to be replaced.
**How to avoid:** Create a NEW screen (`screen-pipeline`) rather than modifying `screen-meeting`. Or: add a toggle button to switch between "transcript view" and "pipeline view" during a meeting. The pipeline view replaces the transcript panel with the Canvas, keeping actions on the right.
**Warning signs:** Existing functionality breaks during testing.

## Code Examples

### Complete Animation Loop Structure
```javascript
// Source: Standard requestAnimationFrame pattern
let animationId = null;
let lastTime = 0;

function animate(currentTime) {
  animationId = requestAnimationFrame(animate);
  const dt = Math.min((currentTime - lastTime) / 1000, 0.1); // clamp delta
  lastTime = currentTime;

  ctx.clearRect(0, 0, canvas.width / dpr, canvas.height / dpr);

  // 1. Draw edges (back layer)
  for (const edge of EDGES) {
    drawEdge(ctx, edge, currentTime / 1000);
  }

  // 2. Draw particles (middle layer)
  updateAndDrawParticles(ctx, dt, currentTime / 1000);

  // 3. Draw nodes (front layer)
  for (const node of NODES) {
    drawNode(ctx, node, currentTime / 1000);
  }
}

function startAnimation() {
  if (!animationId) {
    lastTime = performance.now();
    animationId = requestAnimationFrame(animate);
  }
}

function stopAnimation() {
  if (animationId) {
    cancelAnimationFrame(animationId);
    animationId = null;
  }
}
```

### Edge Drawing with Weight Differentiation (VIZ-03)
```javascript
function drawEdge(ctx, edge, time) {
  const src = nodeById(edge.from);
  const tgt = nodeById(edge.to);
  // Control point offset for curve
  const cx = (src.x + tgt.x) / 2;
  const cy = (src.y + tgt.y) / 2 - 30;

  const isPrimary = edge.weight === 'primary';
  const isActive = edge.activeUntil > time;

  ctx.beginPath();
  ctx.moveTo(src.x, src.y);
  ctx.quadraticCurveTo(cx, cy, tgt.x, tgt.y);

  if (isPrimary) {
    ctx.lineWidth = isActive ? 3 : 2;
    ctx.strokeStyle = isActive ? '#57abff' : '#30363d';
  } else {
    // Informing edge: thinner, dashed
    ctx.lineWidth = isActive ? 1.5 : 1;
    ctx.strokeStyle = isActive ? '#a855f780' : '#30363d60';
    ctx.setLineDash([4, 4]);
  }

  if (isActive) {
    ctx.shadowColor = ctx.strokeStyle;
    ctx.shadowBlur = 6;
  }
  ctx.stroke();
  ctx.setLineDash([]);
  ctx.shadowBlur = 0;
}
```

## Open Questions

1. **Vision events over WebSocket**
   - What we know: Vision results currently come via REST (`/api/frame` returns JSON). The WS connection is audio-only.
   - What's unclear: Should vision pipeline events be sent via WS, or should the frontend infer Camera/Vision activation from REST response timing?
   - Recommendation: Add vision result forwarding to the WS connection. The `session_state` already tracks vision state; emit a pipeline event when `analyze_frame` completes in `api_frame`. Alternatively, the frontend can self-trigger Camera/Vision node activation when it sends a frame and receives a response.

2. **Material Icons in Canvas**
   - What we know: The project uses Material Symbols Outlined web font. Canvas can render web font glyphs via `fillText`.
   - What's unclear: Whether the Material Symbols font will reliably render on Canvas.
   - Recommendation: Use simple emoji or unicode symbols as fallback. Or preload the font, then use `ctx.font = '18px Material Symbols Outlined'; ctx.fillText('mic', x, y)`. Test on Chrome -- if it works, ship it. If not, use colored circles with text labels only.

3. **Pipeline screen vs. modified meeting screen**
   - What we know: Current meeting screen has transcript + actions layout.
   - What's unclear: Should the pipeline view REPLACE the meeting screen or be an additional view?
   - Recommendation: For hackathon demo, REPLACE the meeting screen entirely with the pipeline view. The transcript is less visually impressive than the pipeline graph. Action cards remain on the right side. This avoids toggle complexity.
